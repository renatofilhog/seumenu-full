# Context Mesh Framework

## Overview
Context Mesh organizes project knowledge so humans and AI agents can work from the same source of truth.

The framework follows a simple loop:
1. Intent: document what exists or what needs to be achieved.
2. Build: implement work using approved decisions and known patterns.
3. Learn: capture outcomes, changes, and lessons after implementation.

## Core Separation Rules

### Feature Files
Feature files live in `context/intent/feature-*.md`.

They contain only:
- What the feature does
- Why it exists
- Acceptance criteria

They must not contain:
- Technical implementation details
- Library or framework choices
- File paths or code snippets
- Architecture explanations

### Decision Files
Decision files live in `context/decisions/`.

They contain:
- The technical choice that was made
- Context and constraints
- Rationale
- Alternatives considered
- Outcomes over time

Use decision files for architecture, libraries, integration approaches, deployment choices, and technical workflows.

### Pattern Files
Pattern files live in `context/knowledge/patterns/`.

They contain:
- Reusable implementation guidance
- Code examples
- How and when to apply the pattern
- Real file references that demonstrate the pattern

Patterns explain how to implement work consistently. They do not replace decisions.

### Agent Files
Agent files live in `context/agents/`.

They are optional operational instructions for specialized AI workflows. Do not create them unless there is a clear recurring workflow to codify.

## Plan, Approve, Execute

This pattern is mandatory for non-trivial work:
1. Plan: identify relevant intent, decisions, and patterns before changing code.
2. Approve: confirm or create the decision record for technical direction.
3. Execute: implement the change and then update context.

Do not skip directly from an idea to implementation when the change introduces or alters a technical approach.

## When to Create Files

Create a new feature file when:
- A user-facing or business-facing capability is distinct enough to stand alone
- The capability has separate value, behavior, or acceptance criteria

Create a new decision file when:
- A technical choice affects architecture, maintainability, performance, security, scalability, or team workflow
- A library or integration approach becomes part of the project standard

Create a new pattern file when:
- The codebase contains a reusable implementation structure worth repeating
- Engineers or agents would benefit from a concrete example

Do not create new files when:
- The information is a minor detail better added to an existing file
- A note belongs in the changelog instead of permanent knowledge
- The difference is naming-only and does not represent a new capability, decision, or pattern

## Context Structure

```text
context/
├── .context-mesh-framework.md
├── intent/
│   ├── project-intent.md
│   └── feature-*.md
├── decisions/
│   └── *.md
├── knowledge/
│   ├── patterns/
│   └── anti-patterns/
├── agents/
└── evolution/
    └── changelog.md
```

## Bidirectional Linking Rules
- Feature files must link to related decisions.
- Feature files may link to related patterns.
- Decision files should link back to relevant features.
- Pattern files should link to the decisions they depend on and the features they support.
- Use relative Markdown links only.
- Keep links current when files are renamed or split.

## Status Section Requirements
Every Context Mesh document should end with a `Status` section.

At minimum include:
- Creation date
- Lifecycle status
- Phase when relevant
- A note when the file was generated from existing implementation rather than planned work

## Definition of Done
A context update is complete when:
- Features describe only what and why
- Decisions capture technical choices and rationale
- Patterns show how the codebase implements those choices
- Relevant files are linked bidirectionally
- The changelog reflects the current documented state
- Agent guidance points contributors to the right context before work begins
